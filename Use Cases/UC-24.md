# Use Case 24: Search by Address in the Map UI

**Goal in Context**: A user enters an address in the map interface
search bar so they can quickly navigate to a property and view the
valuation context for that location.\
**Scope**: Property Value Estimator Frontend Map UI\
**Level**: User Goal\
**Primary Actor**: General User\
**Secondary Actors**: Geocoding/Location Resolver Service; Map Rendering
Engine; Suggestion/Autocomplete Service\
**Trigger**: User types an address into the search bar and submits the
search (Enter key or search icon).

## Success End Condition

-   The map centers on the resolved address location, displays a marker,
    and the user can proceed to request an estimate or explore layers
    around that property.

## Failed End Condition

-   The address is not resolved; the map remains unchanged and the user
    receives an error or "no results" message.

## Preconditions

-   User has opened the map UI successfully.
-   Map tiles and UI controls are loaded.
-   Geocoding/Location Resolver service is reachable.
-   User has an internet connection.

## Main Success Scenario

1.  **User** opens the map UI.
2.  **System** displays a map view with a visible address search bar.
3.  **User** begins typing an address (street, city, postal code).
4.  **System** sends partial input to the Autocomplete service.
5.  **Autocomplete service** returns suggested addresses ranked by
    relevance.
6.  **System** displays suggestions in a dropdown list.
7.  **User** selects a suggestion or completes typing and submits the
    search.
8.  **System** sends the full address query to the Geocoding/Location
    Resolver service.
9.  **Geocoding service** resolves the address into a coordinate
    (lat/long) and canonical address format.
10. **System** pans and zooms the map to the returned coordinate.
11. **System** places a marker or highlight at the resolved property
    location.
12. **System** displays the canonical address label and optional context
    (region, neighborhood).
13. **User** can now request an estimate or adjust map filters/layers.

## Extensions

-   **3a**: User enters incomplete input (too short)
    -   **3a1**: System displays "Enter more details" and does not call
        geocoding service.
-   **5a**: Autocomplete returns no suggestions
    -   **5a1**: System displays "No suggestions found" but still allows
        user to submit full query.
-   **7a**: User submits a misspelled or ambiguous address
    -   **7a1**: Geocoding returns multiple candidate results.
    -   **7a2**: System displays candidate list (did you mean?) with map
        previews.
    -   **7a3**: User selects the correct address and system continues
        at step 10.
-   **8a**: Geocoding service is unavailable
    -   **8a1**: System displays "Search unavailable right now" and logs
        error for monitoring.
-   **9a**: Address not found
    -   **9a1**: System displays "No matching address found" and keeps
        map unchanged.
    -   **9a2**: System suggests alternative input formats (postal code,
        city, coordinates).
-   **10a**: User searches an address outside supported coverage area
    -   **10a1**: System displays "Region not supported" warning.
    -   **10a2**: Map optionally pans to nearest supported boundary
        region.

## Related Information

-   **Priority**: Must
-   **Frequency**: Very High
-   **Open Issues**:
    -   Decide whether search supports partial postal code / landmark
        search.
    -   Decide how to rank ambiguous matches (nearest, most populated
        city, etc.).
    -   Determine how map should behave if the address is outside
        available datasets.
