# Use Case UC-11: Compute Commute Accessibility for Work Access Evaluation

**Goal in Context**: Enable the system to compute commute accessibility
metrics for a property so that users can gauge ease of access to
employment centers and workplaces.

**Scope**: Property Value Estimator System

**Level**: Subfunction

**Primary Actor**: Valuation Engine

**Secondary Actors**: Spatial Database; Employment Center Dataset
Service; Routing/Distance Service; Transit Data Service (optional)

**Trigger**: The valuation engine requires commute accessibility
features for a canonical location ID during feature computation.

## Success End Condition

-   Commute accessibility metrics (e.g., travel time to employment
    centers) are computed.
-   A commute accessibility indicator derived from travel metrics is
    available.
-   The computed commute features are attached to the property's feature
    set.

## Failed End Condition

-   Commute accessibility metrics cannot be computed.
-   The system logs the failure and either applies fallback logic or
    omits commute-related features.

## Preconditions

-   A valid canonical location ID has been generated.
-   The canonical location ID can be resolved to geographic coordinates.
-   Employment center dataset (e.g., business districts, major
    employers, transit hubs) is available and indexed.
-   Routing/distance service is operational.
-   Transport mode configuration (e.g., driving, transit, walking) is
    defined.

## Main Success Scenario

1.  The valuation engine receives a canonical location ID for a
    property.
2.  The system retrieves the geographic coordinates associated with the
    canonical location ID.
3.  The system identifies relevant employment centers or commute targets
    based on predefined configuration.
4.  The system invokes the routing service to compute travel time or
    distance from the property to each identified employment center.
5.  The routing service returns commute travel metrics.
6.  The system aggregates commute metrics, such as:
    -   Travel time to nearest employment center\
    -   Average travel time to top N employment centers\
    -   Weighted accessibility index based on multiple centers\
7.  The system applies predefined thresholds or weighting rules to
    derive a commute accessibility indicator.
8.  The system attaches commute metrics and the derived accessibility
    indicator to the property's feature set.

## Extensions

-   **2a**: Canonical location ID cannot be resolved to coordinates
    -   2a1: The system logs the failure.\
    -   2a2: The system omits commute accessibility computation and
        proceeds without work-access adjustment.
-   **3a**: No employment centers configured or found
    -   3a1: The system records default or neutral commute accessibility
        values according to predefined rules.\
    -   3a2: The system logs configuration issue.
-   **4a**: Routing service unavailable or times out
    -   4a1: The system logs routing failure.\
    -   4a2: The system falls back to straight-line (Euclidean) distance
        estimates.\
    -   4a3: The scenario resumes at Step 6.
-   **7a**: Commute weighting configuration missing
    -   7a1: The system applies default commute weighting parameters.\
    -   7a2: The system logs configuration fallback.

## Related Information

-   **Priority**: Should\
-   **Frequency**: Invoked when commute accessibility features are
    enabled in valuation configuration.\
-   **Open Issues**:
    -   Mode-of-transport assumptions (car vs transit vs multimodal).\
    -   Time-of-day variability (peak vs off-peak commute times).\
    -   Inclusion of user-specified workplace destinations.
